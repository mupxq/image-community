# 影像社区 — 产品依赖堆栈

> 从底层基础设施到用户触达的完整依赖分析，标注每一层的成熟度和优先级。

## 第1层：基础设施

这是所有服务运行的底座。

| 组件 | 职责 | 阶段一是否必须 | 备注 |
|------|------|:---:|------|
| 云服务器 | 运行后端服务 | 是 | 阿里云/腾讯云，初期2-4台即可 |
| 对象存储 (OSS/COS) | 存放图片、视频等媒体文件 | 是 | 用户上传+AI生成的图都要存 |
| CDN | 媒体文件加速分发 | 是 | 图片是核心内容，加载速度直接影响体验 |
| 关系型数据库 (MySQL/PostgreSQL) | 用户、作品、关系等结构化数据 | 是 | |
| 缓存 (Redis) | 热数据缓存、会话管理 | 是 | 首页推荐、用户状态 |
| 消息队列 (RabbitMQ/Kafka) | 异步任务处理 | 阶段二 | AI生成任务、通知推送 |
| 搜索引擎 (Elasticsearch) | 作品搜索、标签检索 | 阶段二 | 初期可用数据库模糊查询替代 |

**关键依赖点：对象存储 + CDN 是第一天就必须有的，图片加载慢产品就废了。**

## 第2层：数据层

产品的核心资产。

| 数据域 | 内容 | 核心结构 | 优先级 |
|--------|------|----------|:---:|
| 用户数据 | 账号、资料、偏好 | user表 | P0 |
| 作品数据 | 作品元信息、分镜内容 | work + work_page表 | P0 |
| 创作树关系 | 作品间的分支/续写关系 | parent_work_id + root_work_id 树结构 | P0 |
| 共创贡献 | 每个作品的参与者及角色 | contributor表 | P0 |
| 社交关系 | 关注、粉丝 | follow表 | P1 |
| 消息数据 | 私聊、群聊消息 | conversation + message表 | P1 |
| 书架数据 | 收藏、阅读进度 | bookmark表 | P1 |
| 行为数据 | 浏览、点赞、停留时长 | 事件日志 | P2 (推荐系统依赖) |
| AI训练数据 | 用户创作行为反哺的数据 | 独立数据管道 | P2 |

**关键依赖点：创作树关系是产品最独特的数据结构，work表的 parent_work_id / root_work_id 设计要从第一天就做对，后面改代价极高。**

## 第3层：AI能力层

产品的技术引擎。当前阶段用Mock，但架构上要预留接入口。

| AI能力 | 用途 | 对接方式 | 阶段 |
|--------|------|----------|:---:|
| **图像生成** | 根据文字描述生成分镜画面 | 外部API (Stable Diffusion / DALL-E / 自研) | 阶段一核心 |
| **风格迁移** | 将用户上传的照片转为指定画风 | 外部API或本地模型 | 阶段一核心 |
| **叙事补全** | 根据梗概生成故事大纲和分镜脚本 | LLM API (GPT / Claude / 自研) | 阶段一 |
| **分镜拆解** | 将一段故事自动拆分为多页分镜 | LLM + 后处理逻辑 | 阶段一 |
| **角色一致性** | 同一角色在不同分镜中保持外貌一致 | 图像生成模型的角色锁定能力 | 阶段二 |
| **视频生成** | 从分镜图生成短视频片段 | 外部API (Sora / Runway / 可灵) | 阶段三 |
| **推荐算法** | 个性化内容推荐 | 自建推荐服务 | 阶段二 |
| **内容审核** | 违规内容检测 | 云服务审核API + 自建规则 | 阶段一 |

**关键依赖点：**
- **图像生成和风格迁移是产品能跑起来的前提**，初期可以用第三方API，但要设计好抽象层，后续随时可切换到自研模型
- **角色一致性**是长内容（连载漫画）体验的核心难点，目前业界还没有成熟方案，这可能是阶段二的技术攻坚点
- AI能力层要设计为**可插拔架构**：定义统一的接口，底层可以是任何模型供应商

```
// AI服务抽象接口示例
AIService
├── generateImage(prompt, style) → image
├── transferStyle(image, style) → image
├── generateStoryboard(synopsis, pageCount) → pages[]
├── generateNarrative(context, direction) → text
└── moderateContent(content) → result
```

## 第4层：业务服务层

支撑产品功能的后端服务。

| 服务 | 职责 | 依赖 | 阶段 |
|------|------|------|:---:|
| **用户服务** | 注册登录、资料管理、身份验证 | 数据库、短信/OAuth | P0 |
| **创作服务** | 作品创建、编辑、发布 | 数据库、对象存储、AI能力层 | P0 |
| **共创服务** | 续写/Fork、创作树管理、贡献者追踪 | 创作服务、数据库 | P0 |
| **阅读服务** | 作品浏览、分镜加载、阅读进度 | 数据库、CDN | P0 |
| **书架服务** | 收藏管理、阅读状态、进度同步 | 数据库 | P1 |
| **社交服务** | 关注、粉丝、评论、弹幕 | 数据库、缓存 | P1 |
| **消息服务** | 私聊、群聊、系统通知 | 数据库、WebSocket | P1 |
| **搜索服务** | 作品搜索、用户搜索、标签 | Elasticsearch | P2 |
| **推荐服务** | 首页推荐、相关作品推荐 | 行为数据、推荐算法 | P2 |
| **通知服务** | 推送通知（续写提醒、评论回复等） | 消息队列、推送SDK | P1 |
| **审核服务** | 内容安全审核 | AI审核API | P1 |
| **结算服务** | 打赏、分成计算 | 支付接口、Web3合约 | 阶段三 |

**关键依赖点：共创服务是业务核心，它同时依赖创作服务和数据层的树结构，任何改动都会牵一发动全身。**

## 第5层：前端能力层

用户直接操作的功能模块。

| 模块 | 复杂度 | 依赖 | 阶段 |
|------|:---:|------|:---:|
| **创作编辑器** | 高 | 图片上传、AI接口、分镜排序 | P0 |
| **AI创作向导** | 高 | AI能力层（生成+补全） | P0 |
| **作品阅读器** | 中 | CDN图片加载、手势交互 | P0 |
| **创作树可视化** | 高 | 树结构数据、交互式图形渲染 | P0 |
| **聊天界面** | 中 | WebSocket实时通信 | P1 |
| **书架管理** | 低 | CRUD操作 | P1 |
| **评论/弹幕系统** | 中 | 实时数据同步 | P1 |
| **用户主页** | 低 | 数据展示 | P1 |
| **分享组件** | 中 | 平台分享SDK、生成分享卡片 | P1 |

**关键依赖点：创作编辑器是用户体验的核心，它同时需要支持手动编辑和AI生成两种模式，且要做到流畅切换。**

## 第6层：用户触达层

产品如何到达用户手中。

| 渠道 | 说明 | 阶段 |
|------|------|:---:|
| **iOS APP** | 主力客户端 | P0 |
| **Android APP** | 主力客户端 | P0 |
| **H5/小程序** | 轻量入口，分享链接落地 | P1 |
| **分享卡片/海报** | 社交传播载体 | P1 |
| **创作者后台 (Web)** | 数据分析、作品管理 | 阶段二 |

**关键依赖点：需要确定客户端技术栈——Flutter跨平台 vs 原生双端开发。Flutter初期效率高，但长期可能受限于原生能力（相机、图片处理性能）。**

## 第7层：信任与权益层

产品壁垒所在，但不阻塞初期上线。

| 组件 | 职责 | 依赖 | 阶段 |
|------|------|------|:---:|
| **智能合约** | 共创版权确权、分成规则上链 | 区块链网络（Polygon/Base等L2） | 阶段三 |
| **版权存证** | 作品上链存证，时间戳+哈希 | 智能合约 | 阶段三 |
| **分成结算引擎** | 按创作树贡献比例自动分成 | 智能合约 + 支付接口 | 阶段三 |
| **NFT铸造** | 优质作品/角色的NFT化 | 智能合约 | 远期 |
| **钱包集成** | 用户Web3钱包管理 | 钱包SDK | 阶段三 |

**关键依赖点：Web3层在阶段一二可以先用中心化的分成记录替代，但数据结构要从一开始就记录清楚每个作品链上的贡献关系——这样阶段三上链时可以直接迁移，不需要回溯。**

## 关键依赖链路图

以下是产品最核心的功能"用户续写一个作品"所经过的完整依赖链路：

```
用户点击"续写" (APP)
    │
    ▼
创作编辑器 (前端)  ──── 选择"AI创作"模式
    │                        │
    │                        ▼
    │                  AI能力层
    │                  ├── LLM生成分镜脚本
    │                  └── 图像模型生成画面
    │                        │
    ▼                        ▼
共创服务 (后端)
├── 创建新work记录，parent_work_id指向原作品
├── 继承上游所有contributor
├── 新增当前用户为contributor
├── 存储分镜页面 + 图片上传至OSS
└── 更新创作树索引
    │
    ▼
数据层
├── works表 (树结构)
├── work_pages表 (分镜)
├── contributors表 (共创关系)
└── 对象存储 (媒体文件)
    │
    ▼
通知服务
├── 通知原作者"有人续写了你的作品"
└── 通知同一创作树上的其他共创者
```

## 各阶段依赖优先级总结

### 阶段一（MVP）— 必须解决

| 依赖 | 状态 |
|------|------|
| 云服务器 + 对象存储 + CDN | 需采购 |
| 数据库 (MySQL/PG) + Redis | 需部署 |
| 图像生成API | 需接入第三方 |
| 风格迁移API | 需接入第三方 |
| LLM API (分镜生成) | 需接入第三方 |
| 内容审核API | 需接入 |
| 用户服务 + 创作服务 + 共创服务 | 需开发 |
| 创作编辑器 + 阅读器 | 需开发 |
| APP客户端 (Flutter或原生) | 需开发 |

### 阶段二 — 规模化时解决

| 依赖 | 状态 |
|------|------|
| 消息队列 | 需部署 |
| Elasticsearch | 需部署 |
| WebSocket服务 | 需开发 |
| 推荐算法 | 需开发 |
| 角色一致性AI能力 | 需研发/等待业界方案 |
| 消息服务 + 搜索服务 + 通知服务 | 需开发 |
| H5/小程序 | 需开发 |

### 阶段三 — 生态成型时解决

| 依赖 | 状态 |
|------|------|
| 区块链网络选型 | 需决策 |
| 智能合约开发 | 需开发 |
| 分成结算引擎 | 需开发 |
| 视频生成AI | 需等待业界方案成熟 |
| 创作者后台 | 需开发 |
